Speculative Decoding: Four Tokens for the Price of One

Why LLM generation is slow, what mathematical asymmetry makes a faster approach possible, and how four different techniques exploit it without changing the output at all.

When you type a question into an LLM (large language model the technology behind ChatGPT, Claude, and Gemini), the response appears word by word. Not all at once one token (roughly one word or word-fragment) at a time. Even when the answer is completely predictable, the model still produces each piece in sequence, one step at a time.

This isn't a user interface choice. It's a fundamental constraint of how transformers work. And it turns out to be surprisingly wasteful. This article explains why, what the fix is, and how four different techniques implement that fix in practice.

Why Generating One Token Uses 140 GB of Memory

To generate token number 10, the model needs tokens 1 through 9 to already exist. Before producing each new token, the model builds a rich summary of the entire context so far — and that summary can only be built once the preceding tokens exist. You cannot skip ahead. Every new token depends on all the tokens before it.

So the model runs its full computation once this single run is called a forward pass to produce one token, then feeds that token back as input and runs again. k tokens require k forward passes, in sequence.

Here is what each of those passes actually costs. When the model runs a single forward pass to generate one token, it loads all of its parameters from GPU memory. For Llama-3-70B stored in 16-bit precision (each number using 2 bytes instead of 4, halving memory use), that is 140 GB of data flowing from the GPU's fast on-chip memory chips (high-bandwidth memory, or HBM) to the compute units. The GPU's matrix-multiply units then perform a small amount of math with those weights and produce one token. Then the whole 140 GB must stream again for the next token.

Think of a chef who stores their entire ingredient inventory in a warehouse a ten-minute walk away. Every time they need to make a single cut, they walk to the warehouse, retrieve every inventory list (140 kilograms of paper), carry it back, look up one line, make the cut, and return the paper. The cutting takes two seconds. The walking takes everything else. The chef's hands are idle for 99% of the time.

The GPU's compute units are in the same situation. The arithmetic intensity the ratio of math operations performed to bytes of weight data transferred of a single-token decode is roughly 1–2 FLOPs (floating-point operations) per byte. An A100 GPU needs about 156 FLOPs per byte to keep its compute units busy. The compute units run at less than 2% of their potential capacity. The bottleneck is the memory bus, not the math hardware.

The waste: to produce one token, the GPU loads 140 GB of weights. In the time that memory transfer takes, the compute units could have run 156× more arithmetic. Almost all of that capacity sits idle, waiting for data to arrive.

Before reading on: you have to load 140 GB from memory to produce one token. What if you could get more than one token's worth of useful work out of a single memory load?

Checking Four Tokens Costs the Same as Checking One

Here is the observation that makes speculative decoding possible: generating a sequence of tokens and checking a proposed sequence of tokens are very different operations for a transformer.

Generating k tokens takes k forward passes. You cannot parallelise across them. Each output token is the input to the next step.

Checking whether a proposed sequence of k tokens is correct takes only one forward pass. The transformer's attention mechanism already processes all input positions simultaneously. When you run a forward pass on a sequence of n tokens, you get the model's output distribution at every position at once; this is exactly how prompt processing (prefill) works. A 100-token prompt is processed in one pass, not 100. The same mechanism means you can check k proposed tokens simultaneously: feed the model the existing context followed by the k candidates, run one forward pass, and read off the model's probability for each candidate position at once.

So here is the question: what if a fast, cheap guesser proposed the next k tokens, and then the real model checked all k in one pass? If the guesser is right most of the time, you get k tokens for the cost of roughly one target-model step. Even when it is wrong sometimes, you still come out ahead as long as the guessing is cheap enough.

This is speculative decoding. The cheap guesser is called the drafter. The large model doing verification is called the target model. The proposed tokens are called draft tokens. They are guesses the target model will either accept or correct.

The asymmetry: generating k tokens requires k passes; checking k proposed tokens requires one pass. A cheap drafter plus one verification step can produce multiple tokens for roughly the cost of one target-model call.

A Complete Speculation Round, Token by Token

Let's trace through one full speculation round in detail.

Setup

The current context is "The capital of France is". We'll draft k = 4 tokens per round. We have a drafter of some kind we'll discuss the options shortly and the large target model.

Step 1 Draft

The drafter runs autoregressively, four times. Each step produces one draft token x̃ (notation for a proposed token, as opposed to one sampled directly by the target), which feeds into the next step:

Drafter step 1: sees "...is" → proposes "Paris" (p_draft = 0.82) Drafter step 2: sees "...is Paris" → proposes "," (p_draft = 0.62) Drafter step 3: sees "...Paris ," → proposes "known" (p_draft = 0.48) Drafter step 4: sees "..., known" → proposes "for" (p_draft = 0.71) Draft sequence: ["Paris", ",", "known", "for"]

This costs whatever the drafter costs the point is that it is much cheaper than four full target-model steps.

Step 2 Verify

The target model runs one forward pass on the context plus the four draft tokens. The context's key-value cache (a stored record of prior computation) is already populated from previous steps, so only the 4 new positions need fresh computation. At the end of this single pass, the target model has produced a probability distribution at every position:

Verification pass (only draft suffix is new; context KV is already cached): cached: ["The", "capital", "of", "France", "is"] ← no recomputation needed new: ["Paris", ",", "known", "for"] ← 4 positions computed Position n (after "is") → p_target("Paris") = 0.88 ← verify draft token 1 Position n+1 (after "Paris") → p_target(",") = 0.65 ← verify draft token 2 Position n+2 (after ",") → p_target("known") = 0.14 ← verify draft token 3 Position n+3 (after "known") → p_target("for") = 0.52 ← verify draft token 4 Position n+4 (after "for") → full distribution ← bonus token (see below)

Step 3 Accept or Reject, Left to Right

For each draft token x̃, compute the acceptance probability: 

acceptance_prob = min(1, p_target(x̃) / p_draft(x̃))

Sample u ~ Uniform[0, 1]. If u ≤ acceptance_prob, accept the token and move to the next. Otherwise, reject it, sample a corrected token, and stop all remaining draft tokens are discarded.

Why this formula, not "accept if it matches what the target would have generated"? The target model doesn't always pick the single most likely token it samples from a distribution. A draft token might differ from what the target happened to sample on this run, yet still be a perfectly valid sample from the target's distribution. Rejecting it would discard a valid token unnecessarily.

The formula accounts for this. If p_target(x̃) ≥ p_draft(x̃) the target assigns at least as much probability to the draft token as the drafter did always accept. If the target assigns less, accept proportionally less often. Every token in the output ends up with exactly the probability the target model would have assigned it had it generated everything alone. This property is called lossless: the drafter speeds things up without changing what gets produced.

On rejection, why not just sample from p_target? Let's look at what goes wrong with a concrete three-token example before working out the algebra.

Three-token vocabulary: p_target p_draft min(·,·) deficit = max(0, p_t − p_d) token A: 0.60 0.20 0.20 0.40 token B: 0.30 0.50 0.30 0.00 token C: 0.10 0.30 0.10 0.00 ───── ───── ───── ───── totals: 1.00 1.00 0.60 0.40 Acceptance passes through 0.60 of total mass. P(rejection) = 0.40. If we sample from p_target on rejection: token A: 0.20 (accepted) + 0.40 × 0.60 (corrected) = 0.44 ✗ (target is 0.60) token B: 0.30 + 0.40 × 0.30 = 0.42 ✗ (target is 0.30) token C: 0.10 + 0.40 × 0.10 = 0.14 ✗ (target is 0.10) If we sample from the deficit (normalized) on rejection: Adjusted distribution: token A = 0.40/0.40 = 1.0 token B = 0 token C = 0 token A: 0.20 (accepted) + 0.40 × 1.00 (corrected) = 0.60 ✓ token B: 0.30 + 0.40 × 0.00 = 0.30 ✓ token C: 0.10 + 0.40 × 0.00 = 0.10 ✓

Sampling from p_target gets the probability wrong for every token but in opposite directions depending on the drafter's behaviour. Tokens B and C are overcounted: the drafter was more confident about them than the target (p_draft > p_target), so acceptance already delivered their full target mass; any correction on top pushes them above it. Token A is undercounted: the drafter underestimated it, acceptance only contributed 0.20 of the 0.60 needed, and the p_target correction adds only 0.24 more — not the 0.40 required. The deficit is the only correction that is exactly right for each token. Here is why.

The goal is for token x to appear in the output with exactly p_target(x) probability. It can get there in one of two ways: it is accepted, or the round rejects and x is the corrected sample. Those two contributions must add up to p_target(x).

What acceptance already contributes. The drafter proposes x with probability p_draft(x), and that proposal is accepted with probability min(1, p_target(x) / p_draft(x)). Multiplying: 

P(x accepted) = p_draft(x) × min(1, p_target(x) / p_draft(x))
              = min(p_draft(x), p_target(x))

That is the mass acceptance already delivered. The corrected sample must contribute only the remainder: 

P(rejection) × P(corrected = x | rejection) = p_target(x) − min(p_draft(x), p_target(x))
                                             = max(0, p_target(x) − p_draft(x))

Why this normalises cleanly. Sum the per-token deficits across all tokens: 

Σ_x max(0, p_target(x) − p_draft(x))
  = 1 − Σ_x min(p_draft(x), p_target(x))
  = P(rejection)

The total deficit equals P(rejection) exactly, so dividing each token's deficit by P(rejection) gives a valid distribution that sums to 1. This is the adjusted distribution: 

p_adjusted(x) = max(0, p_target(x) − p_draft(x))  (then normalize)

Back to the example: token B had p_draft(B) = 0.50, already exceeding p_target(B) = 0.30. Acceptance delivers min(0.50, 0.30) = 0.30 for token B its full target mass. The deficit is zero; the correction step must give it nothing. Token A, by contrast, has a deficit of 0.40, which is precisely what remains after acceptance contributes its 0.20.

Applying this to the France example:

Token 1: "Paris" p_target = 0.88, p_draft = 0.82 acceptance_prob = min(1, 0.88/0.82) = 1.0 → ACCEPT Token 2: "," p_target = 0.65, p_draft = 0.62 acceptance_prob = min(1, 0.65/0.62) = 1.0 → ACCEPT Token 3: "known" p_target = 0.14, p_draft = 0.48 acceptance_prob = min(1, 0.14/0.48) = 0.29 Sample u = 0.55 → 0.55 > 0.29 → REJECT Adjusted distribution at this position: "the": p_target=0.45, p_draft=0.05 → max(0, 0.45−0.05) = 0.40 "a": p_target=0.20, p_draft=0.04 → max(0, 0.20−0.04) = 0.16 "also": p_target=0.09, p_draft=0.01 → max(0, 0.09−0.01) = 0.08 "known": p_target=0.14, p_draft=0.48 → max(0, 0.14−0.48) = 0.00 ... Normalize → sample → "the" Token 4: "for" DISCARDED (we stopped at the first rejection)

Result

This round produced three tokens: "Paris", ",", and "the". The next round starts from this updated context.

BASELINE (three tokens, three separate target-model steps): Target pass 1 → "Paris" Target pass 2 → "," Target pass 3 → "the" Cost: 3 × (load 140 GB) SPECULATIVE DECODING (same three tokens): Drafter × 4 steps → "Paris", ",", "known", "for" (cheap) Target pass × 1: ✓ "Paris" accepted ✓ "," accepted ✗ "known" rejected → "the" "for" discarded Cost: 4 × (cheap draft) + 1 × (load 140 GB) If drafter cost ≪ target cost: three tokens for roughly the price of one target step.

Think about it: we discarded "for" because it came after the rejected "known." Why can't we keep it? After all, the target model already computed a probability for "for" at position n+3.

"For" was proposed by the drafter after the sequence "..., known" a context that no longer exists. You've replaced "known" with "the," so the next position is now "..., the". The drafter's proposal for what comes after "known" tells you nothing about what comes after "the." The token must be regenerated from the corrected context.

The bonus token. Notice that the verification pass produced a full probability distribution at position n+4 the position after the last draft token as a free byproduct. If all k draft tokens are accepted, you can sample that distribution for a (k+1)th token at no extra cost. This is why the expected tokens produced per round slightly exceeds k even when the acceptance rate α (the probability that a single draft token is accepted) is high.

How Much Faster? The Expected Speedup Formula

Let α be the per-token acceptance rate, averaged across positions. Assuming each token's acceptance is independent, the expected number of tokens produced per verification step is: 

E[tokens per step] = (1 − α^(k+1)) / (1 − α)

where k = number of draft tokens per round

The formula sums over all outcomes: all k accepted (plus bonus), first rejection at position 1, at position 2, and so on. Each outcome contributes its token count weighted by its probability.

Two things stand out. First, returns diminish with k: going from k=4 to k=8 at α=0.85 adds only ~0.93 expected tokens, not another 3.46. Second, acceptance rate matters far more than draft length: α=0.90 with k=4 beats α=0.60 with k=8. A better drafter is worth more than a longer draft.

But the table assumes the drafter is free. It never is. Let c be the cost of one draft step as a fraction of one target-model step. The realized speedup is: 

Speedup = E[tokens per step] / (k × c + v)
         = [(1 − α^(k+1)) / (1 − α)] / (k × c + v)

where v = cost of the verification step relative to one baseline decode step

At small batch sizes (memory-bandwidth bound):
  v ≈ 1   loading weights once covers all k+1 positions cheaply

At large batch sizes (approaching compute-bound):
  v ≈ k+1   each extra position adds proportional compute; gains disappear

Everything that follows n-gram, draft model, EAGLE, MTP is a different answer to one question: how do you keep c small while keeping α high?

N-gram Drafting: Zero Extra Model Cost

The cheapest possible drafter doesn't run any model at all. It searches the text already in the context window.

Before each draft round, take the last N tokens of the current context the "query n-gram" and search the available text (the original prompt, retrieved passages, previously generated output) for that exact sequence. When you find a match, the tokens that follow it in the source text become the draft. No model runs. No weights load. The drafter cost c ≈ 0.

Example: RAG question answering Retrieved passage in context: "...the Eiffel Tower was constructed between 1887 and 1889..." Context generated so far: "The Eiffel Tower was constructed between" N-gram lookup (n=5, query: "Tower was constructed between"): Match found in the retrieved passage. Tokens that follow in passage: "1887", "and", "1889", "and" Draft: ["1887", "and", "1889", "and"] Target model verifies in one pass: ✓ "1887" accepted ✓ "and" accepted ✓ "1889" accepted ✓ "and" accepted + bonus token Five tokens out. Drafter cost: zero.

N-gram drafting works when the model's output closely quotes or paraphrases its input. Retrieval-augmented generation, summarization, code completion that mirrors surrounding code in these cases, large stretches of the output are literal substrings of the context, and the n-gram lookup fires repeatedly with high acceptance rates.

It fails entirely when the output is novel. Mathematical reasoning, creative writing, any task where the model synthesises rather than recombines input text will produce no n-gram matches. In those cases the drafter falls back to single-token generation and you get no speedup. N-gram drafting is a workload-specific tool, not a universal one.

In vLLM: this is called prompt lookup decoding, enabled via --speculative-config with "method": "ngram". Because c ≈ 0, even a moderate acceptance rate (α ≈ 0.5) produces meaningful speedup on compatible workloads. Check the vLLM speculative decoding docs for current config keys rather than copying flags from older posts.

Draft Model Drafting: A Small Model Does the Guessing

The original speculative decoding papers used a small language model as the drafter. The small model proposes k tokens one at a time, each step feeding its output back as input to the next (autoregressive generation); the large model then verifies all k in one pass.

For this to work, the two models must share the same vocabulary and tokenizer. The acceptance formula compares p_target(x̃) and p_draft(x̃) over the same token ID. If the vocabularies differ even slightly, the comparison is undefined.

Draft model: Llama-3-8B (c ≈ 8/70 ≈ 0.11 per draft step) Target model: Llama-3-70B k = 4 draft steps: 8B step 1 → x̃₁ (cost: 0.11) 8B step 2 → x̃₂ (cost: 0.11) 8B step 3 → x̃₃ (cost: 0.11) 8B step 4 → x̃₄ (cost: 0.11) Total draft cost: 0.44 target steps Verification: 70B forward pass on [context, x̃₁, x̃₂, x̃₃, x̃₄] → 1 target step Total cost per round: 1.44 target steps If α = 0.80 with k = 4: E[tokens] = (1 − 0.8⁵) / 0.2 ≈ 3.36 Speedup = 3.36 / 1.44 ≈ 2.3×

How well this works depends on how much the two models' next-token distributions overlap. Models from the same family trained on similar data Llama-3-8B and Llama-3-70B tend to agree on common continuations, achieving α ≈ 0.70–0.85 on many tasks. Models from different families typically produce lower acceptance rates and less speedup.

Memory footprint: a 70B target (140 GB) paired with an 8B draft (16 GB) requires ~156 GB for weights alone. On A100 80 GB hardware, that spans two GPUs just for weights, leaving almost no room for the KV cache, activations, or runtime overhead. In practice this typically requires four A100s, or weight quantization to reduce the combined footprint enough to leave usable headroom.

EAGLE: Drafting in the Target Model's Own Feature Space

A separate draft model has a fundamental limitation: it knows nothing about what the target model is actually thinking. It learned to predict tokens independently, and the best it can do is guess what the large model might output from the outside.

EAGLE (Li et al. 2024) takes a different approach. Instead of training a separate model from scratch, it trains a single additional transformer block the EAGLE block that operates directly inside the target model's own internal numbers, the high-dimensional vectors the model uses to represent meaning (its feature space).

Why the Target Model's Feature Space Matters

When the target model processes a sequence, it produces a numerical summary of what it has read so far (a hidden state) at every layer. The hidden state just before the final output layer call it f_t is exactly what that output layer (the LM head, short for language model head) transforms into a probability score for every possible next token. If a drafter could predict future values of f_t, it would effectively be predicting what the large model is about to think. That is a much easier problem than predicting tokens from scratch.

There is one subtlety. The hidden state at position t+1 depends not just on the hidden state at t, but on which specific token was sampled at t. Two different sampled tokens produce two different trajectories through the model's internal numbers. EAGLE resolves this by combining the learned numeric representation of the sampled token (its embedding, written e(x_{t+1})) with the hidden state f_t. The embedding tells the EAGLE block which branch of the model's internal space to follow.

EAGLE-1

EAGLE trains a single transformer block that takes (f_t, e(x_{t+1})) as input and outputs an approximate hidden state f̂_{t+1}. That approximation goes through the shared LM head (output layer) to produce the draft probability scores for x_{t+2}. Each EAGLE step costs roughly 1/L of a full model forward pass for a 70-layer model, about 1/70th the cost of one target-model step, far cheaper than running an 8B draft model.

Standard decode step: [x₁,...,x_t] → [Target Model: L layers] → f_t → [LM Head] → x_{t+1} EAGLE draft step (starting from the last accepted position m): f_m is available from the verification pass (the target's true hidden state at m). x̃_{m+1} was already sampled from LM Head(f_m) it comes free. Draft step 1: [f_m ; e(x̃_{m+1})] → [EAGLE Block: 1 layer] → f̂_{m+1} f̂_{m+1} → [Shared LM Head] → x̃_{m+2} Draft step 2: [f̂_{m+1} ; e(x̃_{m+2})] → [EAGLE Block] → f̂_{m+2} → x̃_{m+3} Each EAGLE step costs ≈ 1/L of a target-model step.

Because EAGLE drafts from the target model's own hidden states rather than an independently learned representation, its distributions are closer to p_target. Typical acceptance rates are α ≈ 0.80–0.90, compared to α ≈ 0.70–0.80 for a same-family draft model.

Where f_m comes from: during each verification pass, the target model produces the hidden state at the layer just before the output layer (the second-to-top-layer features, f_m) at every accepted position. These are cached. The EAGLE block always seeds from the target's true f_m at the last accepted position not from its own approximation. The approximation error accumulates only within a single draft window, then gets reset.

EAGLE-2: Adapting the Tree to What Is Being Generated

EAGLE-1 builds a tree of draft candidates rather than a single chain at each depth it expands the top-k most likely tokens, and the target model verifies all branches simultaneously using a special pattern that ensures each branch only attends to its own ancestors, never to sibling branches (a tree attention mask). The longest accepted root-to-leaf path becomes the output. But EAGLE-1's tree has a fixed shape: the same branching factors at every depth, regardless of how hard the current generation is.

EAGLE-2 (Li et al. 2024b) makes the tree dynamic. It estimates a per-node acceptance probability and uses that to allocate the node budget adaptively expanding deep where the model is confident, widening where it is uncertain.

EAGLE-1 (static tree: same shape regardless of context): ┌── x̃₁ₐ ─┬─ x̃₂ₐₐ │ └─ x̃₂ₐᵦ context ──┤ └── x̃₁ᵦ ─┬─ x̃₂ᵦₐ └─ x̃₂ᵦᵦ EAGLE-2 (dynamic tree: shape responds to confidence): When one candidate is clearly likely: ┌── x̃₁ₐ ─┬─ x̃₂ₐₐ ─ x̃₃ₐₐₐ ← go deeper where confident │ └─ x̃₂ₐᵦ context ──┤ └── x̃₁ᵦ (pruned; low confidence) When candidates are equally likely: ┌── x̃₁ₐ ─ x̃₂ₐ context ──┼── x̃₁ᵦ ─ x̃₂ᵦ ← widen rather than deepen └── x̃₁꜀ ─ x̃₂꜀ Same total node budget either way; more of it goes where acceptance is likely.

MTP: When the Model Learns to Draft Itself During Training

All three approaches above treat the target model as fixed and build a drafter around it. Multi-Token Prediction (MTP), as used in DeepSeek-V3 (Liu et al. 2024), takes the opposite approach: change what the model learns during training so that it can draft itself.

The Training Setup

After the main transformer trunk produces its numerical summary of the context (hidden state h_t) at each position t, DeepSeek-V3 adds D lightweight "MTP modules" each one a single transformer layer. MTP Module 1 takes (h_t, embed(x_{t+1})) and predicts x_{t+2}. Module 2 takes the output of Module 1 and predicts x_{t+3}. Each module shares the same final LM head with the trunk. During training, all future tokens are known, so every MTP module receives a clean supervision signal.

Training forward pass at position t (D = 2 MTP modules): x₁,...,x_t → [Main Trunk] → h_t Main head: MTP Module 1: h_t → [LM Head] → p(x_{t+1}) (h_t, embed(x_{t+1})) → [MTP Block 1] → h_t⁽¹⁾ loss: cross-entropy vs x_{t+1} h_t⁽¹⁾ → [LM Head] → p(x_{t+2}) loss: cross-entropy vs x_{t+2} MTP Module 2: (h_t⁽¹⁾, embed(x_{t+2})) → [MTP Block 2] → h_t⁽²⁾ h_t⁽²⁾ → [LM Head] → p(x_{t+3}) loss: cross-entropy vs x_{t+3} Total loss = main loss + λ × MTP-1 loss + λ × MTP-2 loss (DeepSeek-V3 uses λ = 0.1 and D = 1 for the released model)

The MTP loss encourages the trunk to develop hidden states that are useful not just for the next token but for several tokens ahead. This also improves the main model's representations DeepSeek-V3 reports that MTP training improves benchmark performance even when the MTP heads are removed at inference.

At Inference

The main trunk produces h_T and samples x_{T+1} as usual. MTP Module 1 then runs on (h_T, embed(x_{T+1})) and drafts x̃_{T+2}. Module 2 drafts x̃_{T+3}. A verification pass checks both drafts with the standard acceptance-rejection procedure.

Inference at position T: Step 1 Main forward pass: x₁,...,x_T → [Trunk] → h_T → [LM Head] → sample x_{T+1} (definite) (h_T, embed(x_{T+1})) → [MTP Module 1] → sample x̃_{T+2} (draft) (h_T⁽¹⁾, embed(x̃_{T+2})) → [MTP Module 2] → sample x̃_{T+3} (draft) Step 2 Verification pass on [context, x_{T+1}, x̃_{T+2}, x̃_{T+3}]: Position T+1 → p_target(x_{T+2}) → verify x̃_{T+2} Position T+2 → p_target(x_{T+3}) → verify x̃_{T+3} Accept/reject in order. Step 3 MTP modules re-run on newly accepted tokens to draft the next round.

The MTP modules were trained jointly with the trunk they consume, so there is no mismatch between the hidden states (numerical context summaries) they receive at inference and those they trained on. This is the key advantage over EAGLE, which trains its block after the target model is fixed. There is also no separate model checkpoint, no separate KV cache, and no separate drafting call — the modules run as a lightweight tail of the existing forward pass.

The tradeoff: MTP's tight integration requires the model to have been trained with MTP from the start. You cannot attach it to an existing frozen model. EAGLE can be trained and attached to any frozen model post-hoc. MTP is more efficient; EAGLE is more flexible.

Five Things That Surprise People

1. "Doesn't speculative decoding change the output?"

No. The acceptance-rejection procedure is provably lossless. The output distribution of the full system target model plus drafter plus acceptance step is identical to the distribution you'd get by running the target model alone. This was proved independently by Leviathan et al. (2023) and Chen et al. (2023). The draft tokens are just proposals; the target model has final authority over every token that appears in the output.

2. "Why discard everything after the first rejection?"

Because later draft tokens were generated conditioned on the rejected token being in the context. Draft token 4 was proposed after the sequence "..., known" but you've now replaced "known" with "the." Token 4's proposal is built on a context that no longer exists. There is no way to salvage it without re-running the drafter from the corrected position.

3. "Can I use any small model as a drafter?"

Only if it shares the target model's exact vocabulary and tokenizer. The acceptance formula compares p_target(x̃) and p_draft(x̃) over the same token ID. If the vocabularies differ, the token IDs don't correspond and the comparison is meaningless. Beyond vocabulary, acceptance rates drop significantly when the two models' distributions don't overlap well — same-family models trained on similar data tend to agree far more than models from different families or training distributions.

4. "Higher acceptance rate always means more speedup, right?"

Not necessarily. The speedup also depends on the drafter cost c. A drafter that achieves α = 0.95 but costs 0.5 target steps per token might produce less actual speedup than one that achieves α = 0.80 at c = 0.01. Both matter: 

Speedup = [(1 − α^(k+1)) / (1 − α)] / (k × c + v)

This is why n-gram drafting (c ≈ 0) is competitive even with moderate acceptance rates on the right workloads.

5. "Speculation should help at any batch size, right?"

Wrong and this is where many deployments go wrong. At batch size 1–16, the GPU is memory-bandwidth-bound: loading weights once covers all k+1 positions cheaply, so the verification step costs roughly the same as one baseline decode step.

At large batch sizes (32+), the GPU becomes compute-bound. The verification step now processes (k+1) × batch_size positions proportionally more computation. The denominator in the speedup formula grows with k while the numerator stays bounded. At large enough batch sizes, speculation reduces throughput rather than improving it. Always measure before deploying at batch size > 16.

Measure on your own workload. Published speedups are measured on specific models, hardware, batch sizes, and task distributions. α varies enormously across tasks. A 3× speedup on a benchmark chat dataset tells you little about your specific RAG pipeline or code assistant. Measure α and end-to-end latency on a representative sample of your real traffic before committing to a drafter choice.

Key Takeaways

  • LLM generation at small batch sizes is memory-bandwidth-bound: each token requires loading the full model's weights, leaving the compute units nearly idle.
  • Verification is cheaper than generation: checking k proposed tokens takes one forward pass; generating k tokens takes k.
  • The acceptance-rejection procedure is lossless: the combined output distribution is identical to running the target model alone, proved by the adjusted distribution sampling on rejection.
  • The four approaches trade off drafter cost c vs. acceptance rate α: n-gram (c≈0, workload-dependent α), draft model (moderate c, moderate α), EAGLE (low c, high α, post-hoc), MTP (lowest c, high α, requires pretraining).
  • Speculation is most useful at batch sizes 1–16. At large batch sizes the verification cost grows proportionally and the gains disappear.
