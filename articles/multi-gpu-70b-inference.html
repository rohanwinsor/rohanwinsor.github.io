Two Workloads, One Cluster: Multi-GPU 70B Inference From First Principles

A ground-up explanation of tensor, pipeline, and data parallelism; quantization; and the real-time vs. batch tradeoff.

This article is inspired by Horace He's Making Deep Learning Go Brrr From First Principles, which explains GPU performance from first principles using a factory analogy. That framing carries forward here; we just add more factories.

Scope: this is a first-principles design analysis, not an original benchmark report. The memory calculations are theoretical, and the configurations are candidates to test against a defined workload and SLO. The core thesis is to size one replica first, then replicate that shape for capacity. Official runtime documentation and source papers are linked where they inform the reasoning.

There is a class of models that won't fit on a single GPU at all. Llama-3-70B in FP16 or BF16 takes about 140 GB for weights alone. An A100 has 80 GB. The model doesn't fit. You need at least two GPUs just to load the weights. But fitting one model replica is only the first problem. You must also decide how much hardware each replica should use and how many independent replicas to run. Those choices change the balance between per-request latency, aggregate throughput, and cost.

This article is organized around that tension. We'll work through tensor and pipeline parallelism for distributing one replica, data parallelism for adding replicas, and quantization for reducing the memory needed by each replica. Then we'll turn those mechanisms into candidate configurations for interactive and offline workloads.

We will not cover: model training, FSDP or DeepSpeed ZeRO, expert parallelism for MoE models, or cross-datacenter deployment. All are real topics; covering them here would dilute the focus.

Weight memory ≈ num_parameters × 2 bytes
Llama-3-70B ≈ 70,000,000,000 × 2 = 140 GB

The KV cache headroom column matters, but reading it requires one more step. For Llama-3-70B the full FP16/BF16 KV cache costs 320 KiB per token (2 × 80 layers × 8 KV heads × 128 dim × 2 bytes); under tensor parallelism, KV heads are sharded across the TP ranks (each rank is one GPU in the parallel group) just like weight matrices. Llama-3-70B has 8 GQA KV heads, so at TP=8 each GPU holds exactly 1 KV head: 40 KiB per token per GPU. The effective capacity is the total headroom across all GPUs divided by the full 320 KiB/token cost: 

TP=2 capacity: (2 × 10 GB) ÷ (320 KiB/token × 2,048 tokens/session) ≈ 30 concurrent 2K sessions
TP=4 capacity: (4 × 45 GB) ÷ (320 KiB/token × 2,048 tokens/session) ≈ 268 concurrent 2K sessions
TP=8 capacity: (8 × 62.5 GB) ÷ (320 KiB/token × 2,048 tokens/session) ≈ 745 concurrent 2K sessions
  (theoretical upper bound before runtime overhead)

At TP=8 each GPU holds only 17.5 GB of weights, leaving ~62.5 GB free on every GPU. Because each token's KV is sharded across all 8 ranks, that per-GPU headroom adds up into one shared pool: 8 × 62.5 ≈ 500 GB effective, not the 62.5 GB a single GPU appears to offer. That pooled capacity is what produces the ~745-session figure above. Runtime overhead (NCCL buffers, CUDA graphs, activation memory) will reduce these numbers in practice.

More GPUs buy memory capacity and can reduce compute time, but they introduce a new cost that didn't exist before: inter-GPU communication. The next sections separate the three ways GPUs can participate in a deployment.

Tensor Parallelism: The Horizontal Split

Tensor parallelism (TP) is the strategy that splits each weight matrix across N GPUs. Every GPU participates in every forward pass, working on its slice of the computation in parallel.

The Mechanism

In the classic Megatron-style pattern, TP uses two complementary layer types.

Column-parallel layers (attention QKV projections, FFN up-projection): the weight matrix is split along the output dimension, so each GPU holds a column shard. Every GPU receives the same input and independently computes a shard of the output. The results remain partitioned for the following row-parallel layer, so this projection does not require an immediate reduction. This independence comes directly from which dimension is split: each output column of the result depends only on the corresponding weight columns — never on another GPU's slice. GPU 0 evaluates x @ W[:,0:d/4], GPU 1 evaluates x @ W[:,d/4:d/2], and so on; these are fully disjoint column ranges of the same matrix multiply.

Column-parallel layer (TP=4), no communication needed: Every GPU gets the SAME full input x and holds a column shard of W: [GPU 0] [GPU 1] [GPU 2] [GPU 3] x (full) x (full) x (full) x (full) W[:,0:d/4] W[:,d/4:d/2] W[:,d/2:3d/4] W[:,3d/4:d] ↓ ↓ ↓ ↓ out[0:d/4] out[d/4:d/2] out[d/2:3d/4] out[3d/4:d] Output stays split and feeds straight into the row-parallel layer. No AllReduce in this layer.

Row-parallel layers (attention output projection, FFN down-projection): the weight matrix is split along the input dimension. Each GPU receives its shard of the prior layer's output and computes a partial sum of the final output. These partial sums must be summed across all GPUs via an AllReduce before the next layer can begin. This assignment works because the shard boundaries match: the column-parallel layer split the weight along the output dimension (e.g. columns 0:d/4 on GPU 0), and the row-parallel layer splits along the input dimension — the same d/4 slice, now viewed as an input row range. GPU 0's output from the first layer is exactly the input rows GPU 0 needs for the second layer, so no intermediate communication is required between the two.

Row-parallel layer (TP=4), where AllReduce happens: Each GPU holds a shard of the input (from the prior column-parallel layer): [GPU 0] [GPU 1] [GPU 2] [GPU 3] x[0:d/4] x[d/4:d/2] x[d/2:3d/4] x[3d/4:d] W[0:d/4,:] W[d/4:d/2,:] W[d/2:3d/4,:] W[3d/4:d,:] ↓ ↓ ↓ ↓ partial_0 partial_1 partial_2 partial_3 ← partial sums └──────────────┴──────────────┴──────────────┘ AllReduce (sum) (every GPU gets the full output) ↓ Complete output → next layer

The ordering is deliberate. A column-parallel layer feeds a row-parallel layer, and the activation between them is element-wise, so each GPU keeps working on its own shard with no communication until the single AllReduce at the end of the pair. The non-linearity is the key constraint: element-wise functions like GeLU or SiLU applied to a shard produce the same result as applying them to the full tensor and then taking the shard — so each GPU can run the activation independently. Reversing the order breaks this. If the up-projection were row-parallel (first layer) and the down-projection column-parallel (second layer), each GPU would hold a partial sum after the first layer, not a true shard. Because GeLU is non-linear, GeLU(partial_sumi) ≠ the correct contribution — so you would need an AllReduce to assemble the full intermediate before applying the activation. That forces two reductions per block instead of one:

Why column-parallel comes BEFORE row-parallel FFN block: Z = GeLU(X · A) · B Split the first matrix A by COLUMNS, the second matrix B by ROWS: A = [ A1 │ A2 ] B = [ B1 ] [ B2 ] [GPU 0] [GPU 1] X·A1 X·A2 ← column-parallel: no communication GeLU GeLU ← element-wise activation, so each GPU handles its own shard alone ·B1 ·B2 ← row-parallel: each GPU holds a partial └───────┬────────┘ AllReduce (sum) ← the ONE sync, at the very end ↓ Z → next block

In this simplified dense decoder block, each transformer layer contains one column-parallel → row-parallel pair for attention and another for the FFN, giving two reductions per layer. For Llama-3-70B's 80 layers, that is 160 reductions per forward pass. Exact collectives can differ when a runtime uses sequence parallelism, fused operations, or architecture-specific kernels.

The Conference Call Cost

An AllReduce is a coordination step: all ranks exchange partial results, and dependent computation cannot continue until the reduction is available. In the factory analogy: after each production stage, all factories have a mandatory conference call to combine their partial work before the next stage can start.

The cost depends on where the TP group sits:

  • NVLink/NVSwitch: the preferred boundary for TP because collectives stay on a high-bandwidth local fabric
  • PCIe: workable at smaller TP sizes, but communication can erase the compute savings
  • Across nodes: possible with InfiniBand and GPUDirect RDMA, but every layer now pays network latency

Collective algorithms, topology, contention, and message size determine the actual cost. Increasing TP degree can stop helping or make latency worse, so topology must be measured rather than inferred from GPU count alone.

The point: TP is a latency lever, not a free speedup. It divides matrix multiplication across GPUs but adds communication to every layer. The useful rule is: choose the smallest TP degree that fits the model and satisfies the latency SLO. Beyond that point, extra ranks may add more collective overhead than useful compute.

Pipeline Parallelism: The Vertical Split

Pipeline parallelism (PP) takes a completely different approach: rather than splitting each weight matrix, it splits the model by layers. GPU 0 holds layers 0–19, GPU 1 holds layers 20–39, GPU 2 holds 40–59, GPU 3 holds 60–79. A forward pass flows sequentially through the pipeline: GPU 0 computes layers 0–19, then passes the intermediate activations to GPU 1, which computes layers 20–39, and so on.

Pipeline Parallelism (PP=4, Llama-3-70B): [GPU 0] [GPU 1] [GPU 2] [GPU 3] Layers 0–19 Layers 20–39 Layers 40–59 Layers 60–79 │ │ │ │ forward ─activations→ forward ─activations→ forward ─activations→ forward ↓ output token

The Communication Advantage

In the basic schedule, PP requires one point-to-point transfer per stage boundary: the activation tensor from one GPU to the next. For Llama-3-70B with PP=4, that's 3 transfers per forward pass, each roughly the size of one stage-boundary activation: tokens_in_microbatch × d_model × bytes_per_element. These are less frequent point-to-point messages, a pattern that is often easier to place across nodes than the repeated collectives required by TP.

This is why PP is a common strategy when one model replica must span multiple nodes. It is not automatically the best way to use every multi-node cluster: if a complete replica fits on each node, independent replicas avoid cross-node communication during the forward pass.

The Pipeline Bubble Problem

PP has a fundamental efficiency problem when processing a single request. While GPU 1 is computing layers 20–39, GPU 0 is idle; it already finished its layers and has nothing to do until the next request arrives. For a single in-flight request with four equally balanced stages and no stage overlap, average utilization is 25%. The other three factories in the assembly line are dark.

Pipeline Parallelism (PP=4), single request: Time → [GPU 0] ████░░░░░░░░░░░░░░░░░ (busy 25% of the time) [GPU 1] ░░░░████░░░░░░░░░░░░░ [GPU 2] ░░░░░░░░████░░░░░░░░░ [GPU 3] ░░░░░░░░░░░░████░░░░░ ← output ████ = computing ░ = idle (pipeline bubble)

The Theoretical Fix: Micro-Batching

M is the number of micro-batches the global batch is divided into. A micro-batch is one contiguous slice: with a global batch of 256 sequences and M=4, each micro-batch holds 64 sequences. The scheduling rule is that each stage, as soon as it finishes a micro-batch and forwards its activations downstream, immediately starts the next one rather than waiting for the full batch to drain.

This creates a three-phase schedule:

  • Startup (N−1 steps): Stage 0 begins immediately on micro-batch ①. Stage 1 can only start once it receives Stage 0's activations — one step later. Stage 2 must wait one more step, and so on. The pipeline fills one stage per step for N−1 steps.
  • Steady state (M−N+1 steps, when M≥N): all N stages are simultaneously busy with different micro-batches. This is the only phase where every GPU contributes.
  • Drain (N−1 steps): Stage 0 finishes its last micro-batch and goes idle. The pipeline empties front to back, mirroring the startup exactly.

The combined bubble — idle slot-steps from startup and drain — is N(N−1), independent of M. Adding micro-batches extends only the steady-state middle without touching the startup or drain. When M ≈ N the steady-state window is nearly absent and the bubble dominates; M must significantly exceed N to amortize the fixed overhead.

Three-phase pipeline schedule (PP=4) M=4 (startup=3, steady=1, drain=3), total steps = 7: [Stage 0] ①②③ ④ ░░░ [Stage 1] ░①② ③ ④░░ [Stage 2] ░░① ② ③④░ [Stage 3] ░░░ ① ②③④ Bubble = N(N−1) = 12 idle slot-steps (fixed; does not grow with M). Useful = M×N = 16. Total = 28. Utilization = 16/28 = 4/7 ≈ 57%. With M=N the steady state collapses to one step; startup and drain dominate. ───────────────────────────────────────────────────────────────────── M=10 (startup=3, steady=7, drain=3), total steps = 13: [Stage 0] ①②③ ④⑤⑥⑦⑧⑨⑩ ░░░ [Stage 1] ░①② ③④⑤⑥⑦⑧⑨ ⑩░░ [Stage 2] ░░① ②③④⑤⑥⑦⑧ ⑨⑩░ [Stage 3] ░░░ ①②③④⑤⑥⑦ ⑧⑨⑩ Same bubble = N(N−1) = 12. Useful = M×N = 40. Total = 52. Utilization = 40/52 = 10/13 ≈ 77%. More micro-batches dilute the same fixed bubble over more useful work.
Theoretical utilization ≈ M / (M + N − 1)

where M = number of micro-batches, N = pipeline depth (number of stages)

PP=4, M=4:   4 / (4 + 3) ≈ 57%
PP=4, M=10:  10 / (10 + 3) ≈ 77%
PP=4, M=27:  27 / (27 + 3) = 90%

Implementation caveat: this is the ideal GPipe-style utilization model, not a claim that a particular vLLM release or executor overlaps stages exactly this way. Autoregressive dependencies, continuous batching, stage imbalance, and scheduler behavior all affect the realized bubble. Verify overlap with an execution trace and pin the runtime version when reporting results.

In this ideal model, approaching 90% utilization with PP=4 requires 27 micro-batches in the pipeline run. Real serving does not map request count to micro-batches this cleanly, but the direction is useful: a shallow queue exposes bubbles, while abundant independent work gives the scheduler more opportunity to overlap stages. The practical amount of work needed must be established for the exact runtime and request distribution.

The takeaway: PP solves a placement problem first: it lets one replica span devices or nodes by assigning different layers to different stages. Pipeline utilization can improve with more independent work, but PP should still be compared with smaller TP groups and data-parallel replicas whenever a complete replica fits on fewer GPUs.

A Common Hybrid When One Replica Must Span Nodes

When a model must span multiple nodes, there are three candidate topologies for two 8× A100 SXM4 nodes:

  • TP=16 across all 16 GPUs: every layer is split 16 ways, which fits in memory. But all 160 AllReduces per forward pass must cross InfiniBand. The highest-frequency communication in the cluster lands on the slowest link.
  • PP=2 with one GPU per stage: one GPU per node holds its layer range; the remaining 14 GPUs sit idle. Memory fits; hardware utilization does not.
  • Hybrid — TP=8 within each node, PP=2 across nodes: all 160 AllReduces per forward pass stay inside each node's NVLink fabric. Only a single stage-boundary activation tensor crosses InfiniBand per forward pass. High-frequency traffic on the fast link; one low-frequency transfer on the slow one.
tensor_parallel_size=8    # AllReduces stay within each node's NVLink fabric
pipeline_parallel_size=2  # one activation tensor crosses InfiniBand per forward pass
                          # Node A: layers 0–39,  Node B: layers 40–79

The hybrid wins by matching communication frequency to link speed, not because TP or PP alone cannot shard the model. For Llama-3-70B — 140 GB, which fits on one node at TP=8 (17.5 GB per GPU against an 80 GB limit) — the hybrid is unnecessary. It becomes a genuine requirement when a model is large enough that TP=8 per node still cannot fit each stage's weights (roughly 1 T+ parameters in FP16). For 70B, independent per-node replicas are the right starting point; the hybrid is a candidate to benchmark against them.

Data Parallelism: Replicate the Model

Tensor and pipeline parallelism answer one question: how should a single model replica be distributed? Data parallelism (DP) answers another: how many independent replicas should serve the request stream?

Each DP replica owns a full copy of the model, although that copy may itself be sharded with TP or PP. Requests are routed across replicas, and dense-model replicas process independent batches without synchronizing every transformer layer. DP therefore increases aggregate request capacity, but it does not make one request's forward pass faster.

When the model fits on one node, DP and TP face the same topology tradeoff across nodes. Extending to TP=16 (spanning two 8-GPU nodes) keeps a single replica but puts all 160 AllReduces per forward pass on InfiniBand. DP=2 with TP=8 per node keeps every AllReduce inside each node's NVLink fabric, with zero cross-node communication during forward passes. DP buys both the right communication topology and twice the request capacity.

Two nodes, one TP=8 replica per node (DP=2, TP=8): Request router ├──► Replica 0: Node A [G0 G1 G2 G3 G4 G5 G6 G7] TP=8 └──► Replica 1: Node B [G0 G1 G2 G3 G4 G5 G6 G7] TP=8 No model-layer collective crosses the node boundary. Each replica has its own weights, scheduler, and KV-cache pool.
Strategy What is split? Primary benefit Primary cost
Tensor parallelismEach layer's matricesFit a replica; potentially lower per-request latencyCollectives throughout every forward pass
Pipeline parallelismLayer rangesFit one replica across devices or nodesPipeline bubbles and stage-boundary transfers

The sizing rule: first find the smallest TP/PP shape that fits one replica and meets its latency target. Then use remaining GPUs for DP replicas. Larger TP can still be correct, but it should earn its extra GPUs by improving the measured latency or throughput SLO.

Quantization: Fitting More with Less

If you can't or don't want to use more GPUs, quantization is the other lever. It shrinks the model's memory footprint by representing weights in a lower-precision format.

Quality degradation varies by model, method, calibration data, and task. Test on your own evaluation set before committing.
Format Weight memory (70B) Minimum GPUs (A100 80GB) Typical quality cost
FP16 / BF16~140 GB2 (tight) / 4+ (more headroom)Baseline
INT8 weight-only or W8A8~70 GB plus scales/metadata1 (tight) / 2+ (more headroom)Workload- and kernel-dependent; evaluate quality

FP8~70 GB plus scales/metadataBackend-dependentFormat matters: W8A8 and weight-only paths have different hardware support

Quantization has a secondary effect beyond memory. Low-batch autoregressive decode is often memory-bandwidth-bound because each step streams a large fraction of the weights while doing relatively little work per weight. Smaller weights can reduce that pressure. As batch size grows, arithmetic intensity rises and compute or kernel efficiency can become the bottleneck, so a 2× reduction in weight bytes should not be presented as a guaranteed 2× speedup.

The practical choice for 70B: select a concrete format and kernel, not just a bit width. For example, FP8 W8A8 is not the same deployment path as weight-only FP8, and current vLLM support differs by GPU architecture. On A100 specifically, do not assume native FP8 W8A8 support. Check the vLLM compatibility table, measure throughput on the target GPU, and run task-level quality evaluations before committing.

Two Workloads, Different Objectives

The useful distinction: TP and PP determine the shape of one replica; DP determines how many independent replicas process the queue. The workload determines which resource deserves the next GPU.

Real-Time Inference: Optimize for Latency

The scenario: users are waiting for responses. They care about time-to-first-token (TTFT) and time per output token (TPOT). A slow response is a bad product experience regardless of how efficiently you used the GPU.

The starting point: the smallest replica that meets the SLO. TP can reduce per-request compute time by making GPUs collaborate on each layer, but scaling is sublinear and can reverse when collectives dominate. Once a replica meets the TTFT and TPOT targets, additional GPUs are often more useful as DP replicas that reduce queueing delay and absorb more requests.

Factory analogy: TP assigns more workers to one order; DP opens another complete service line. If an order itself is too slow, add workers to that line. If orders are waiting in the lobby, another line may help more.

Key configuration choices for real-time:

  • Keep one TP group within a fast-interconnect domain when possible; crossing a slow boundary adds collective latency to every layer
  • Tune max_num_seqs against the latency SLO; more batching can raise throughput while worsening per-request latency
  • Generous KV cache budget: users have long conversations; don't preempt them mid-session
  • Add DP replicas when queueing dominates; replicas increase service capacity without widening per-layer collectives
  • Use PP when placement requires it, not merely because multiple nodes are available

Batch Inference: Optimize for Throughput

The scenario: you're processing offline workloads: annotating a dataset, running evaluations, generating synthetic data. No user is waiting. A request that takes 10 minutes is fine. What matters is cost: how many tokens can you generate per dollar?

The starting point: large batches plus as many right-sized replicas as fit. Continuous batching keeps each replica busy. If a 70B replica fits within one node, DP lets multiple nodes process independent batches with no layer-by-layer cross-node communication. PP becomes necessary when one replica cannot fit within the chosen device group; it is not inherently a throughput upgrade over replicas.

Factory analogy: it's an overnight warehouse run. You have 10,000 orders to fill. Individual order latency is irrelevant; optimize for orders per hour. Keep several complete production lines busy. Use a multi-stage line only when one product is too large for a single building.

Key configuration choices for batch:

  • High max_num_seqs: limits how many sequences are scheduled per iteration; set high enough to keep the GPU continuously saturated
  • High max_num_batched_tokens: process more tokens per step to amortize overhead
  • Chunked prefill: batch prompts may be long; chunk them to keep one long prefill from monopolizing a scheduler iteration
  • Prefer DP across self-contained replicas when the model fits each replica's device group
  • Use TP within a fast-interconnect domain and PP across nodes only when one replica must span them
  • Tighter KV cache: request lengths are predictable from the queue; you can tune headroom more precisely

The Mental Model: Which Configuration Are You In?

The choices above boil down to a handful of questions. Work through them in order:

Question Answer → Implication
Does 70B FP16 fit on my available GPUs? No → increase TP/PP degree, or use a supported quantization format after quality evaluation
What is the smallest TP/PP group that fits one replica? Start there; memory feasibility sets the lower bound before performance tuning
Does that replica meet the TTFT and TPOT SLO? No → test a larger TP group on the fastest available interconnect
Does queueing dominate end-to-end latency or is aggregate throughput the goal? Yes → add DP replicas before widening a replica that already meets its SLO
Can one complete replica fit within each node? Yes → compare independent per-node replicas against any cross-node PP design
Must one replica span multiple nodes? Yes → TP within fast local domains and PP across the slower node boundary is a common candidate
Do my GPUs connect through PCIe rather than NVLink/NVSwitch? Benchmark TP scaling carefully; topology and peer-to-peer support may set the practical limit
Am I changing precision? Pin the exact format, kernel, GPU support, and task-level quality result

These modes are not mutually exclusive. A production service can use TP inside every replica, DP across replicas, and PP only for replicas that must span placement boundaries. The same cluster can also use a latency-oriented replica shape during business hours and a throughput-oriented shape for queued jobs, provided model-loading and operational costs make reconfiguration worthwhile.

Putting It All Together

Let's trace a concrete 70B deployment on a realistic cluster: 8× A100 80GB SXM4 connected through NVSwitch, running Llama-3-70B in FP16 or BF16.

Memory Layout (TP=8)

  • Weights per GPU: 140 GB ÷ 8 = 17.5 GB
  • KV cache headroom per GPU: 80 − 17.5 ≈ 62.5 GB
  • KV cache capacity (cluster-wide): under TP=8 each GPU holds 1 KV head (40 KiB/token). The 8 GPUs collectively store all 8 heads for every cached token, so nominal capacity = 8 × 62.5 GB ÷ 320 KiB/token ≈ 1.5 million cached tokens
  • Concurrent 2K-context sessions (theoretical): 500 GB total ÷ (320 KiB × 2,048) ≈ 745 across the whole TP group, not 745 per GPU. Actual capacity is lower after runtime reservations, fragmentation, active-token workspace, and safety margin.

Real-Time Mode

Configure for interactive use: low concurrency, fast per-user response. 

data_parallel_size = 1
tensor_parallel_size = 8
pipeline_parallel_size = 1
max_num_seqs = 32

This is a latency-oriented candidate: all 8 GPUs collaborate on each request. It is justified only if TP=8 materially improves tail TTFT or TPOT over TP=4 while meeting the expected load. If TP=4 already meets the SLO, two DP replicas of TP=4 may reduce queueing and serve more users.

Batch Mode

Reconfigure for overnight processing: high concurrency, maximize throughput. 

data_parallel_size = 2
tensor_parallel_size = 4
pipeline_parallel_size = 1
max_num_seqs = 256  # per replica; tune to the request-length distribution

This candidate divides the node into two independent TP=4 replicas. Each replica has roughly 45 GB of nominal non-weight HBM per GPU, and the two schedulers can process independent batches. It is not guaranteed to beat TP=8: smaller matrix shards, duplicated runtime reservations, topology, and reduced per-replica batch size can change the result.

Adding a second identical node creates two natural candidates: DP=2, TP=8 for two latency-oriented replicas, or DP=4, TP=4 for four smaller replicas. PP=2 enters the candidate set when one replica must span both nodes; it should not be enabled merely because the second node exists.

The Underlying Principle

Multi-GPU inference introduces a new resource that didn't exist in single-GPU deployments: inter-GPU communication bandwidth. It sits alongside compute and memory bandwidth as a potential bottleneck, and it interacts with your workload type in a specific way. TP pays communication throughout every forward pass in exchange for a wider replica. PP communicates at stage boundaries and introduces pipeline scheduling concerns. DP duplicates the replica and its memory footprint in exchange for independent request capacity.

The right configuration is the smallest replica shape that satisfies memory and per-request latency, multiplied into enough replicas to satisfy arrival rate and throughput, with quantization considered only alongside kernel support and quality. The hardware topology generates constraints; workload measurements settle the choice.
