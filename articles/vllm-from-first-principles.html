vLLM From First Principles: How the LLM Factory Learned Not to Waste

A ground-up explanation of PagedAttention, prefix caching, and continuous batching.

This article is directly inspired by Horace He's excellent Making Deep Learning Go Brrr From First Principles. That article explains why your GPU kernel is slow. This one picks up where it leaves off and asks: given that you understand the hardware, why is your LLM server still slow? The factory analogy from that article carries forward; we just add a warehouse.

Version scope: implementation-specific details were reviewed against vLLM v0.22.1 and its stable documentation in June 2026. vLLM evolves quickly, so treat backend names and defaults as versioned details rather than timeless properties of the architecture.

Consider an 80 GB NVIDIA A100. It offers roughly 2 TB/s of memory bandwidth and 312 TFLOPS of BF16 tensor math. On paper, it should be able to serve tokens at a serious clip. In practice, naive LLM serving squanders most of that. Two independent inefficiencies in how memory is managed and how work is scheduled are responsible.

This article follows those two inefficiencies from the underlying hardware constraints to vLLM's fixes. By the end you'll have a clear mental model of how PagedAttention improves KV-cache memory management, how prefix caching avoids repeated prefill work, and how continuous batching keeps useful work flowing through the GPU.

We will not cover: multi-GPU tensor parallelism, quantization, or API-level usage. Those are real topics, but they'd dilute the ideas here.

The Factory, Revisited

The brrr article set up the GPU-as-factory metaphor: the GPU's compute units are the factory floor, DRAM is the warehouse, and the memory bus is the supply chain connecting them. The central insight there was that moving data between the warehouse and the factory floor is often the bottleneck, not the factory itself.

We're going to extend that metaphor. In LLM serving, the factory floor (the GPU compute) has to work with a warehouse (GPU HBM) that holds per-request working state. The problem is that the warehouse management and job scheduling are both terrible by default.

The Two Phases of LLM Inference

A standard decoder-only text-generation request has two conceptual phases, and they are different enough to behave like separate workloads.

Phase 1: Prefill

You send in a prompt. The model reads the whole thing in a single forward pass: all tokens processed at once, in parallel.

That parallelism is what makes prefill compute-bound: processing all N positions together means large, dense matrix multiplications across the full input. Attention is O(n²) in sequence length, so a longer prompt is meaningfully more expensive, but it's all done in one shot. The factory floor is running at capacity.

The output: the first generated token, sampled from the model's probability distribution at the last position.

Phase 2: Decode

Now the model needs to generate the rest of the response, one token at a time. Token T must be generated before token T+1 can even start, because T+1's attention depends on T. This serialization is baked into the autoregressive structure of transformers; it's not a software limitation.

Each decode step takes the single most-recently-generated token as input, runs a full forward pass through all the model's layers, and produces the next token. To generate one token, the model still has to load all of its weights through the memory bus. For Llama-3-8B in FP16, that's 8 billion parameters × 2 bytes ≈ 16 GB of data moving from HBM to the compute units per output token. At the A100's 2 TB/s peak bandwidth, that's an idealized lower bound of about 8 ms of memory traffic per token for a single sequence; in practice, compute and memory transfers overlap in hardware pipelines, and batching multiple sequences amortizes the weight loading across all of them. Even so, decode at batch size 1 runs at roughly 1–2 FLOPs/byte. The A100's ridge point sits at ~156 FLOPs/byte (312 TFLOPS / 2 TB/s): anything below that is memory-bandwidth-bound, not compute-bound. Decode is 100× below that line. Classic memory-bandwidth-bound territory, the same bottleneck the brrr article covers at the kernel level.

Prefill: Input: [tok₁ tok₂ tok₃ ... tokₙ] ← all tokens processed in ONE step Output: [tok_{n+1}] Bottleneck: COMPUTE (big matrix multiplications) Throughput: fast per-token, GPU saturated Decode: Step 1: [tok_{n+1}] → tok_{n+2} Step 2: [tok_{n+2}] → tok_{n+3} Step 3: [tok_{n+3}] → tok_{n+4} ... ← N steps for N output tokens Bottleneck: MEMORY BANDWIDTH (16 GB of weights per step, for Llama-3-8B FP16) Throughput: slow per-token, compute underutilized (waiting on the bus, not the ALUs)

A typical response of 200 tokens = 1 prefill step + 200 decode steps. The factory floor spins up to full speed briefly, then grinds through 200 tiny jobs, loading the entire warehouse contents for each one.

The KV Cache

During decode, each new token must attend over every previous token to compute its output. You have two options: recompute the key and value vectors for all prior tokens from scratch on every step (very expensive), or cache them (sensible). The cache of all previously computed key and value tensors is the KV cache.

For each token, across all layers, you store: 

KV bytes per token =
  2            (key + value)
× num_layers   (32 for Llama-3-8B)
× num_kv_heads (8)
× head_dim     (128)
× dtype_bytes  (2 for FP16)
= 131,072 bytes ≈ 128 KB per token

A sequence currently containing 2,048 tokens uses 256 MB of KV cache. The model weights themselves take about 16 GB, giving an upper bound of ~64 GB from an 80 GB A100 before accounting for activations, CUDA graphs, allocator state, and other runtime memory. That upper bound is enough for around 500,000 cached tokens in theory. But with naive allocation, that budget disappears much faster than it should.

Waste #1: The Memory Fragmentation Problem

Given that KV cache is expensive, how does a naive serving system manage it? The obvious approach: when a request arrives, pre-allocate a contiguous block of GPU memory for its KV cache, sized to the maximum possible sequence length. You need it contiguous because the attention kernels are written to assume a flat, contiguous layout.

This is a disaster for two reasons.

Internal fragmentation

You don't know how long the completed sequence will be. If you reserve KV space for 4,096 token positions and the sequence finishes after using only 42, about 99% of that reservation was unused. The floor section is roped off; nobody else can use it; and most of it is empty.

External fragmentation

Even if every request were exactly the right length, contiguous allocation causes holes. After several requests complete and free their allocations, the free memory is scattered in gaps. A new request that needs 1,000 tokens of contiguous space can't get it (even if 1,400 tokens of free memory exist) because no single gap is large enough.

Measured result: The vLLM paper (SOSP '23) found that under this naive allocation strategy, 60–80% of reserved KV cache memory goes to waste due to fragmentation. Your GPU is mostly holding empty reservations.

The warehouse analogy makes this visceral:

Imagine a warehouse where every incoming order, before a single parcel arrives, gets its own section of floor roped off, sized for the largest order that could ever come in. Most orders are small. Most of the floor sits empty. New orders can't get started because there's no space, even though 80% of the floor is technically unoccupied, just held by reservations that may never fill.

The Fix: PagedAttention

The insight behind PagedAttention is simple: don't pre-allocate. Manage the warehouse with fixed-size shelf units, assigned on demand.

Instead of one big contiguous region per request, divide GPU HBM into small, fixed-size physical blocks managed in a single global block pool shared across all requests. Using a 16-token block as a concrete example, each block holds KV tensors for 16 token positions. The block size is configurable and can depend on the platform and attention backend. For Llama-3-8B in FP16, this example works out to 16 × 128 KB ≈ 2 MB per block.

Each in-flight request gets its own block table: a small ledger that maps "logical block 0 → physical block #47, logical block 1 → physical block #103, ..." All requests draw their physical blocks from the same shared pool, but each request's block table is private to it. The blocks don't need to be contiguous. The block table is the indirection layer that makes non-contiguous storage look contiguous to the attention kernel.

Blocks are allocated as the sequence grows. Starting from an empty cache, a sequence containing 42 total KV positions (prompt plus generated tokens) uses 3 blocks (⌈42/16⌉), with at most 15 empty slots in the final partial block. In a real request, decode tokens first consume any space left in the prompt's final block. Worst-case waste: the last block is nearly empty. The vLLM paper reports that paging keeps this internal waste below 4% in its design, down from 60–80% under the compared systems.

In the original paper, PagedAttention reduced internal KV-cache waste from the 60–80% reported for the compared systems to below 4%. Across its tested OPT-13B–175B workloads, vLLM delivered 2–4× the throughput of Orca and FasterTransformer at comparable latency.

Bonus: Copy-on-Write for Beam Search

Beam search keeps N candidate continuations alive simultaneously, all sharing the same prompt prefix. Under naive allocation, you'd copy the entire KV cache for each beam, like photocopying the whole order ledger every time you branch.

With PagedAttention, all beams share the parent's blocks by reference. The block table for each beam points to the same physical blocks, like multiple order ledgers pointing to the same shared shelf section. A block only gets physically copied when a beam actually diverges and needs to write new KV data into it. Pruned beams simply drop their ledger entries; once no sequence references a block, it can return to the free pool without copying the shared prefix.

Bonus: Prefix Caching

Many production workloads share a system prompt: "You are a helpful assistant. Today's date is..." Under naive allocation, every request has to recompute the KV vectors for this shared prefix from scratch on every prefill. PagedAttention supplies the fixed-size block substrate that makes sharing practical, but it does not automatically turn every block into a reusable cache entry. vLLM's separate automatic prefix caching feature hashes completed blocks and looks them up when a later request begins with the same prefix.

vLLM uses a forward hash chain: each full block's cache key includes its token IDs, its parent block's hash, and any extra values needed to distinguish contexts such as LoRA adapters, multimodal inputs, or cache salts. When a new request arrives, vLLM walks forward through its prompt blocks, looking up each block's hash in the cache. It reuses every matching leading block and stops at the first miss. Chaining makes each key represent the entire prefix leading to that block. Hashing is still a security and correctness boundary because finite hashes can collide. vLLM exposes hash-algorithm choices, including SHA-256 variants, and supports cache salts for isolation. The prefix-caching design document covers those tradeoffs and cache isolation.

This is the warehouse equivalent of a shared "standard kit" shelf: common items that go into most orders are stocked once and referenced from multiple order ledgers. For workloads with heavy shared prefixes, this can remove substantial repeated prefill work. The gain depends on the reusable prefix length, cache-hit rate, prompt-to-output ratio, and whether the workload is prefill-bound. Prefix caching does not accelerate the decode tokens that follow the shared prompt.

Waste #2: The Batching Problem

Even with perfect memory management, there's a separate problem: GPU idle time. The issue is how requests are traditionally grouped into batches.

Static batching works like this:

  1. Collect N requests
  2. Run forward passes until the longest sequence in the batch finishes generating
  3. Return all N results
  4. Start the next batch

The problem is step 2. Imagine a batch of 32 requests where 30 of them generate 50-token responses and 2 of them generate 500-token responses. The 30 short requests finish in one-tenth of the time. But the GPU keeps running, still doing real work for the 2 long requests, while the 30 completed slots sit idle, producing nothing.

The result: under mixed-length workloads, the GPU spends a significant fraction of its time running forward passes for just a handful of long stragglers while most batch slots sit empty. The factory floor has 32 stations. 30 finished an hour ago. 2 are still running. The other 30 stations are empty and dark.

The Fix: Continuous Batching

The model runs in a tight loop. Each iteration is one GPU forward pass, and the scheduler decides which requests are in that pass, reconsidering the instant any request finishes rather than waiting for a whole batch to drain.

Before that loop makes sense, two facts about a single request need to be on the table.

When does a request finish? Decode emits one token at a time, but something has to end it. Two things can. The model can sample an end-of-sequence token, a special token in the vocabulary that means "stop", drawn from the probability distribution exactly like any ordinary token. Or the request can hit a length limit the caller set when they sent it ("generate at most N tokens"). Until one of those fires, the request stays in the running set and produces one more token on every pass.

How much can one pass do? A forward pass carries a fixed token budget: the total number of tokens it will compute that iteration, summed across every request in the batch. A decode step spends just one token per running request, since each produces a single new token. Prefill is far heavier: ingesting a 1,000-token prompt costs 1,000 tokens of budget at once. The budget is a tuning knob, set high enough to keep the GPU saturated and low enough to bound how long any single step takes. It is what gates admission: the scheduler can only pull in new work while the step still has budget, and free KV blocks, to spare.

With those two facts in hand, each forward pass does three things, in order:

  1. Collect completions. Scan the running set. Any request that just emitted its end-of-sequence token or reached its length limit is finished. Remove it, release its KV blocks back to the global pool, and stream the result to the caller.
  2. Admit from waiting. With the freed capacity, pull the next request off the waiting queue and add it to the running set. Three constraints gate admission: the step's remaining token budget, a cap on how many requests may run at once, and available KV blocks. If block allocation fails mid-admission, the scheduler may preempt a lower-priority running request instead: release its KV blocks, reset its computed-token count, and return it to the front of waiting. Prefix caching may recover that work on re-admission; otherwise the request restarts.
  3. Run the next forward pass on the updated running set. Every request advances by one token. Return to step 1.

This is iteration-level scheduling. The batch never "finishes" as a unit. Tokens flow out as requests complete, new requests flow in immediately, and the GPU never idles waiting for a straggler.

Chunked Prefill: Solving Head-of-Line Blocking

Continuous batching creates a subtle new problem. Prefill is compute-intensive: processing a 32,000-token prompt in a single step takes a long time, during which all the ongoing decode steps are blocked. A single long-context request can cause a latency spike for every other user.

Chunked prefill fixes this by splitting large prefills according to the scheduler's token budget and interleaving them with decode work. Instead of one enormous prefill step that blocks the factory floor for a second, you do N small prefill steps, each taking a normal amount of time, mixed in with the regular flow of decode work. The configured budget trades inter-token latency against time-to-first-token and throughput; values such as 2,048 tokens are tuning points, not a universal chunk size. See vLLM's optimization guidance.

The Mental Model: Which Bucket Are You In?

The brrr article ends with a diagnostic framework for GPU kernels: is your bottleneck compute, memory bandwidth, or overhead? Each answer points to a different class of optimization. The same discipline applies to LLM serving, but the bottlenecks are different.

Symptom Likely bottleneck Evidence to collect Candidate levers
GPU runs out of memory; can't serve more concurrent users even though GPU isn't busy Memory capacity KV-cache usage approaches its configured limit; admissions stall or preemptions rise; model weights and runtime allocations leave little room for KV blocks PagedAttention; shorter limits; weight quantization to free HBM; KV-cache quantization to shrink the cache directly
GPU utilization is low while requests are queued; throughput is below the expected operating point Scheduling or batch-shape underutilization Running-token count and batch occupancy stay low; a timeline shows gaps, stragglers, or long prefills delaying decode work Continuous batching; tune token and sequence budgets; chunked prefill; inspect CPU and networking overhead before blaming the GPU
Per-token latency is high even with good GPU utilization Decode bandwidth, kernels, or communication A profiler shows high HBM traffic or communication time; TPOT changes with batch size, quantization, or parallelism strategy Weight quantization; kernel/backend tuning; reconsider tensor-parallel topology

Use this table to generate hypotheses, not final diagnoses. The same symptom can have several causes: low utilization may come from request arrival rate, CPU tokenization, synchronization, or networking rather than batching; high TPOT may come from collectives or an inefficient attention backend rather than raw HBM bandwidth. Measure the queue, cache, batch, and profiler signals before choosing a lever. Also keep the two forms of quantization distinct: weight quantization reduces model memory and can indirectly leave more HBM for KV blocks, while KV-cache quantization reduces bytes stored per cached token.

PagedAttention and continuous batching together are the core of what makes vLLM fast. The vLLM paper measured 2–4× higher throughput at comparable latency versus Orca and FasterTransformer across OPT-13B through OPT-175B on ShareGPT and Alpaca workloads. Those gains come from fitting more useful state in memory and keeping the available batch capacity occupied.

Diagnose before treating. Applying prefix caching to a workload with no shared prefixes wastes implementation effort. Know which bucket you're in.

Mapping the Model to vLLM's Source

The architectural story above becomes more concrete in vLLM v0.22.1's V1 implementation. The scheduler's own comment explicitly says there is no separate "decoding phase" or "prefill phase" in the scheduler. Instead, each request has a computed-token frontier, and schedule() assigns enough work to move that frontier toward the request's known prompt and output tokens. 

token_budget = max_tokens_for_this_step

for request in running:
    work = min(tokens_needed_to_catch_up(request), token_budget)
    blocks = allocate_slots(request, work)
    if blocks are unavailable:
        preempt a lower-priority request and retry

for request in waiting:
    cached_prefix = get_computed_blocks(request)
    work = min(remaining_known_tokens(request), token_budget)
    admit request only if its KV slots can be allocated

That pseudocode is intentionally simplified, but it preserves the control flow worth noticing:

  • Scheduler.schedule() computes token work for running requests under a step-wide budget and asks the KV-cache manager for slots. The waiting-request path later checks cached prefixes and admits new or resumed work under the remaining budgets.
  • KVCacheManager.get_computed_blocks() asks for the longest eligible cache hit, while allocate_slots() accounts for cached, newly computed, external, and lookahead tokens before allocating blocks.
  • generate_block_hash_extra_keys() and hash_block_tokens() show the prefix hash chain directly: the parent hash, current block tokens, and contextual keys such as LoRA, multimodal data, and cache salt contribute to the next block hash.
  • BlockPool.touch() and free_blocks() implement sharing with reference counts. A cache hit increments references; releasing a request decrements them, and zero-reference blocks become eviction candidates.
  • Scheduler._preempt_request() frees a request's KV blocks, resets its computed-token frontier, and returns it to waiting. This is the concrete cost behind the word "preemption."

Reading these paths also exposes the tradeoffs hidden by the clean warehouse metaphor. Paging adds block-table metadata and requires attention kernels that understand indirect storage. Prefix caching spends CPU work on hashing and only reuses full eligible blocks. Continuous batching needs admission, fairness, and preemption policies. Chunked prefill improves decode responsiveness but changes the tradeoff among time to first token, inter-token latency, and throughput. The mechanisms are useful because those costs are smaller than the waste they remove for the target workload, not because they are free.

Benchmarking the Claims

The numerical results above come from cited papers and project benchmarks; they are not original measurements from this article. To test whether the same ideas help your workload, hold the model, GPU, software version, input distribution, output length, and sampling settings constant, then vary one serving feature at a time.

vLLM's serving benchmark CLI reports the metrics that matter: request and token throughput, time to first token (TTFT), time per output token (TPOT), and inter-token latency. A minimal benchmark shape is: 

vllm serve MODEL [configuration-under-test]

vllm bench serve \
  --backend openai \
  --model MODEL \
  --dataset-name random \
  --input-len 1024 \
  --output-len 256 \
  --num-prompts 1000 \
  --request-rate RATE \
  --max-concurrency CONCURRENCY \
  --save-result \
  --save-detailed

Run warm-ups first, repeat each configuration, and report median plus tail latency rather than one favorable run. Sweep concurrency instead of quoting a single requests-per-second number: an optimization that improves latency at concurrency 1 may reduce maximum throughput under load. Record the exact vLLM commit or release, model revision, GPU, driver, CUDA version, command line, and raw JSON results so another engineer can reproduce the chart.

Putting It All Together

Let's trace a request through a well-configured vLLM instance to see how these pieces interact:

  1. Request arrives → enters the waiting queue. If automatic prefix caching is enabled, the scheduler checks for matching blocks from this prompt's prefix. If yes, those KV blocks are reused, avoiding the corresponding repeated prefill computation.
  2. Prefill → the prompt is processed (possibly chunked to avoid blocking decodes). Physical KV blocks are allocated as needed via PagedAttention.
  3. Scheduling and model loop → each iteration, the scheduler runs the running requests it can fit under the current token and KV budgets; this may include prompt chunks or decode tokens. New blocks are allocated as each request's KV cache grows.
  4. Completion → the sequence reaches EOS or another stopping condition. Its block references are released. Unreferenced blocks become reclaimable; with prefix caching, reusable blocks may remain cached until the allocator needs to evict them. The scheduler can admit more waiting work at the next scheduling opportunity.

The warehouse avoids large per-request reservations. Under sustained load, the scheduler can refill capacity as sequences finish instead of waiting for an entire static batch.

That is the core vLLM story: PagedAttention turns the KV cache into manageable blocks, prefix caching reuses blocks that represent repeated prompt prefixes, and continuous batching keeps useful work flowing as requests arrive and finish. Together, they attack the memory waste and scheduling waste that make a naive LLM server slow.
